# Source type guide

Use this file when the incoming link is not obviously a short post or a normal article.

## Short-form post / note

Typical signals:
- Xiaohongshu / Rednote / social share link
- sparse page text
- strong reliance on captions or images

Approach:
- use title, description, hashtags, visible cues
- do not pretend to have a full transcript
- emphasize practical takeaways over exhaustive reconstruction

## Long-form article / essay

Typical signals:
- clear headline and body text
- byline / date / sections
- enough text to summarize directly

Approach:
- capture thesis, structure, major arguments, practical implications
- preserve the author’s framing without copying large chunks

## Product / engineering / documentation page

Typical signals:
- release post, engineering blog, API or docs page
- structured headings, diagrams, lists, examples

Approach:
- summarize design choices, workflow, constraints, tradeoffs
- include implementation implications when useful
- separate source claims from your own takeaways

## Partial-access pages

If browser/fetch gets only part of the page:
- summarize accessible parts only
- state the limitation briefly
- avoid filling gaps with confident guesses
